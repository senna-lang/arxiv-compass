## Background

現状、全バッチは手動実行。自動化することで arXiv新聞を完全自律運用にする。

現在のバッチ構成:

| 頻度 | スクリプト | 役割 |
|---|---|---|
| 日次 | `fetch_daily.py` | arXiv最新論文を取得・スコアリングして `data/YYYYMMDD.json` を生成 |
| 週次 | `recommend.py` | ratings + map.json からおすすめ論文を生成 |
| 月次 | `map.py` | 全論文をBERTopicでクラスタリングして `map.json` を再生成 |

バッチ実行後は Cloudflare Pages への再デプロイが必要（静的ビルドのため）。

**制約**:
- specter2_base（440MB）はローカルGPU/CPUが必要 → クラウドWorkerでは動かない
- デプロイは `wrangler pages deploy dist/` → Cloudflare API Token が必要
- arXiv は月〜金のみ更新（土日は skip）

## Spec

### 自動化方式: GitHub Actions

ローカル launchd より GitHub Actions が望ましい理由:
- マシンの起動状態に依存しない
- ログが GitHub UI で確認できる
- Cloudflare API Token / ratings_url をシークレットで管理できる
- PR ベースのデプロイと分離できる

**ただし specter2_base の embedding が問題**。GitHub Actions の無料枠（2GB RAM）では OOM になる可能性がある。
→ モデルキャッシュ戦略 or セルフホストランナーの検討が必要。

### ワークフロー設計

```
.github/workflows/
  daily.yml     # 月〜金 09:00 JST（00:00 UTC）
  weekly.yml    # 月曜 09:30 JST（00:30 UTC）
  monthly.yml   # 毎月1日 10:00 JST（01:00 UTC）
```

#### daily.yml（日次）

```yaml
on:
  schedule:
    - cron: '0 0 * * 1-5'   # 月〜金 09:00 JST
  workflow_dispatch:          # 手動実行
```

ステップ:
1. `pip install -r requirements.txt`
2. モデルキャッシュ（`actions/cache` で `~/.cache/huggingface`）
3. `python scripts/fetch_daily.py --log`
4. `npm run build`
5. `npx wrangler pages deploy dist/`
6. 生成した `data/YYYYMMDD.json` を git commit & push

#### weekly.yml（週次）

```yaml
on:
  schedule:
    - cron: '30 0 * * 1'   # 月曜 09:30 JST
```

ステップ:
1. `npm run sync:ratings`（KV から ratings.json を取得）
2. `python scripts/recommend.py --log`
3. `npm run build`
4. `npx wrangler pages deploy dist/`
5. `data/recommendations.json` を git commit & push

#### monthly.yml（月次）

```yaml
on:
  schedule:
    - cron: '0 1 1 * *'   # 毎月1日 10:00 JST
```

ステップ:
1. `python scripts/map.py` （時間かかる）
2. `recommend.py` も再実行（map.json が変わるため）
3. ビルド & デプロイ
4. `data/map.json` を git commit & push

### 必要なシークレット

| シークレット | 用途 |
|---|---|
| `CLOUDFLARE_API_TOKEN` | wrangler deploy |
| `CLOUDFLARE_ACCOUNT_ID` | wrangler deploy |
| `RATINGS_KV_NAMESPACE_ID` | sync:ratings |

### requirements.txt

現状 requirements.txt が存在しない。GitHub Actions 用に作成が必要。


## Key Decisions

- Decision: GitHub Actions を採用。Rationale: ローカルマシン依存を排除し、ログとシークレット管理を一元化。
- Decision: HuggingFace モデルキャッシュを actions/cache で保持。Rationale: specter2_base（440MB）を毎回ダウンロードすると遅い＆コスト増。
- Decision: バッチ結果（JSON）を git commit して push。Rationale: Astro の静的ビルドは data/ の JSON をビルド時に読むため、リポジトリに含める必要がある。
- Decision: 月〜金のみ日次実行。Rationale: arXiv が土日更新しないため無駄な実行を省く。

## Notes

- GitHub Actions 無料枠: 2,000 分/月。daily=5分×20日=100分、weekly=10分×4=40分、monthly=30分×1=30分 → 余裕あり
- specter2_base の RAM: 推論だけなら 4〜6GB。GitHub Actions の標準ランナーは 7GB → ギリギリ動く可能性あり。要検証。
- `data/` は `.gitignore` に入れていない前提（現在コミット済みのため）
- デプロイ後に `wrangler pages deployment list` でステータス確認するステップを追加すると運用しやすい
