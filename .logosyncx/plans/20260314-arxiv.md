## Background

現在のarXiv新聞は自分専用にカスタマイズされたシステム（config.jsonにinterest_profile/categories固定）。
これを誰でも使えるマルチユーザーサービスに拡張したい。

現状の構成:
- Astro + Cloudflare Pages（静的ビルド + Workers API）
- Cloudflare KV: ratingsを1ユーザー分保存
- Pythonバッチ: parse.py（日次）、map.py（月次）、recommend.py（週次）
- specter2_base（440MB）で論文embeddingとスコアリング

## Spec

### ユーザーフロー
1. OAuthでアカウント作成（Google / GitHub）
2. 初期設定: arXivカテゴリ選択 + interest_profile（自然言語）入力
3. 日次: 自分のカテゴリ×interest_profileに基づくスコアリング済み論文が届く
4. 星でランキング（現状と同じ）
5. 週次レコメンド・月次地図（現状と同じロジック、per-user）

### 変更が必要なコンポーネント
- **DB**: KV → Cloudflare D1（users / profiles / ratings / recommendations テーブル）
- **Auth**: Clerk.dev or Auth.js（CF Workers対応）
- **バッチパイプライン**: embedding/scoringの分離が必須

## Key Decisions

**Decision: paper embeddingとuser scoringを分離する。**
Rationale: specter2（440MB）のロードをユーザー数分繰り返すとGitHub Actionsの無料枠が破綻する。
「カテゴリ別に論文取得→embed（1回/日/カテゴリ）」と「per-userのdot product scoring（安価）」を分ける。

**Decision: DBはCloudflare D1を採用する。**
Rationale: CF Workers上でSQLiteがネイティブ動作。KVと違いユーザー別クエリができる。
無料枠（5GB、10M reads/day）は〜200ユーザー規模で十分。

**Decision: 地図（map.json）はグローバル共有とする。**
Rationale: BERTopicは全論文を対象とするため月次1回の生成で全ユーザーが共有可能。
クラスタリングはカテゴリでフィルタせず全カテゴリ統合で行う。

## Notes

### スケール感
| ユーザー数 | バッチ | インフラコスト |
|---|---|---|
| 〜50人 | GitHub Actions無料枠で余裕 | ほぼ無料 |
| 〜200人 | Actions有料 or 小VPS（$5〜） | 月$5〜10 |
| 200人超 | バッチ設計の見直しが必要 | 要検討 |

### 実装順序（推奨）
1. D1スキーマ設計（users / profiles / ratings / paper_embeddings / recommendations）
2. バッチパイプライン分離（embed_papers.py + score_user.py に分割）
3. Auth追加（Clerk推奨: CF Workers対応が最も簡単）
4. フロントエンド: per-userページ + onboarding UI

### 既存コードの再利用
- compute_score(), _cosine_similarity() → そのまま流用可
- BERTopicパイプライン → グローバルmap生成として流用可
- Astro APIルート → per-user対応に拡張

### 未解決の問題
- specter2 query adapter（adhoc_query）の導入タイミング（マルチユーザー化と同時にやるか）
- 新規ユーザーのコールドスタート: interest_profileのみでスコアリング（現行と同じ）で開始
- 論文の重複: 複数ユーザーが同じカテゴリを選択した場合のembed共有キャッシュ設計
