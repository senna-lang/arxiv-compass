## Background

arXiv新聞のAstroアプリ（静的生成 + Cloudflare Pages）のフロントエンドパフォーマンスを改善する。現状、各日付ページは175KB(raw)/29KB(gzip)。依存はastroのみでフレームワークなし。主なボトルネックはHTML内のデータ重複、ratings APIの非効率な全件取得、レンダリング最適化の欠如。

## Spec

5つの改善を実施する:

1. **abstract の data 属性重複を除去**: `.star-rating` div の `data-abstract` を削除し、クリックハンドラでDOM走査に変更。約26KB/ページ削減。
2. **ratings API をページ単位フィルタに変更**: `GET /api/ratings?ids=paper1,paper2,...` でページ内のpaper_idだけ取得。`ids`なしは全件返す（recommend.py互換）。クライアントJSはページ内paper_idを収集してクエリに渡す + `requestIdleCallback` で遅延実行。
3. **`content-visibility: auto`**: ファーストビュー外のカードのレンダリングを遅延。
4. **map iframe の lazy load**: `loading="lazy"` 追加。
5. **preconnect + meta description**: `<link rel="preconnect" href="https://arxiv.org" />` と `<meta name="description">` 追加。

## Key Decisions

Decision: ratings API は ids パラメータでフィルタリング対応、パラメータなしは全件返す。Rationale: recommend.py のバッチ取得との後方互換性を維持しつつ、フロントエンドのレスポンスサイズを削減。

Decision: abstract取得はDOM走査 `card.querySelector("details p")` を使用。Rationale: 全3コンポーネント（PaperCard, RecommendSection, SerendipitySection）で `article > details > p` 構造が統一されている。

Decision: content-visibility は CSS のみの変更。Rationale: 非対応ブラウザでは無視されるだけなのでフォールバック不要。
